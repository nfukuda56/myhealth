# 減量管理システム DB要件定義（v3）

> 改訂履歴：v3（2026-06-05）技術基盤を GitHub Pages + Supabase 構成へ全面改訂

---

# 1. システム概要

本システムは、減量・健康管理における行動ログを記録し、身体変化との関連性を可視化することを目的とする。

単なるカロリー計算ではなく、

* 食事
* 運動
* 身体計測
* コメント
* タグ

などを時系列イベントとして保持し、ユーザー自身の行動認知・フィードバックへ繋げることを重視する。

---

# 2. システム目的

## 主目的

* 日々のカロリー収支可視化
* 体重推移把握
* 行動と結果の関連性把握

---

## 副目的

* 継続入力しやすい記録基盤
* コメントやタグを含むライフログ化
* 将来的分析拡張
* AI画像解析連携余地

---

# 3. 技術基盤

| 項目          | 内容                              |
| ----------- | ------------------------------- |
| フロントエンド     | GitHub Pages（静的サイトホスティング）       |
| フロントエンド実装   | HTML / CSS / JavaScript（バニラまたはVue等） |
| バックエンド      | Supabase（BaaS）                  |
| DB          | PostgreSQL（Supabase管理）          |
| DB接続        | Supabase JavaScript Client（supabase-js） |
| 認証          | Supabase Auth                   |
| API         | Supabase Auto REST API / RPC    |
| タイムゾーン      | Asia/Tokyo                      |
| 文字コード       | UTF-8                           |

---

## 3.1 構成概要

```text
GitHub Pages（静的フロントエンド）
  │
  │  supabase-js（HTTPS）
  ▼
Supabase
  ├─ Auth（認証・セッション管理）
  ├─ PostgreSQL（データ永続化）
  ├─ Auto REST API（CRUD）
  ├─ RPC（集計・複雑クエリ）
  └─ Row Level Security（データ保護）
```

---

## 3.2 Supabase利用方針

| 機能              | 利用方針                          |
| --------------- | ----------------------------- |
| Auto REST API   | 基本CRUD操作に使用                   |
| RPC（DB関数）       | 集計・複数テーブル横断クエリに使用             |
| Row Level Security | 全テーブルに適用。ユーザーは自分のデータのみ参照・操作可能 |
| Realtime        | 将来拡張時に検討                      |
| Storage         | 将来の画像解析連携時に使用予定               |

# 4. データ管理思想

## 基本思想

本システムでは、

```text
「推定値の絶対精度」
より
「継続観測による傾向把握」
```

を重視する。

---

## rawデータ保持方針

入力値は加工せず rawデータとして保持する。

集計値や分析結果は、別VIEWまたは別集計テーブルとして後付け生成する。

---

## イベントログ設計

身体計測・食事・運動はすべてイベントとして扱う。

各イベントは measured_at を保持し、時系列分析可能とする。

---

# 5. 日付管理仕様

## 日次単位

入力UIは「その日」単位とする。

---

## 日付締め

1日の切替時刻は AM4:00 とする。

---

## 管理項目

| 項目          | 内容        |
| ----------- | --------- |
| target_date | 論理日       |
| measured_at | 実際のイベント日時 |

---

## 例

| target_date | measured_at      |
| ----------- | ---------------- |
| 2026-05-29  | 2026-05-30 01:20 |

AM4締めのため同一日扱い。

## 複数計測

# 16. 認証方針

## 基本方針

Supabase Auth を使用する。

初期段階はメールアドレス＋パスワード認証を採用する。

---

## user_id 管理

全テーブルの user_id は Supabase Auth の `auth.users.id`（UUID）と紐付ける。

アプリ独自の users テーブルは `auth.users` を参照するプロファイルテーブルとして保持する。

---

## Row Level Security（RLS）

全テーブルに RLS を有効化する。

基本ポリシー：`auth.uid() = user_id` の行のみ参照・操作可能。

```sql
-- RLSポリシー基本形（各テーブルに適用）
CREATE POLICY "ユーザー自身のデータのみ操作可"
  ON テーブル名
  FOR ALL
  USING (auth.uid() = user_id);
```

---

# 17. PostgreSQL設計方針

## 主キー

`BIGSERIAL` を基本とする。

ただし Supabase Auto REST API との親和性を考慮し、将来的に `UUID` 移行も検討する。

---

## user_id 型

`UUID` 型を使用する。（Supabase Auth の `auth.uid()` に合わせる）

---

## 数値型

PFC・体重は `NUMERIC` を使用する。

---

## タイムスタンプ

`TIMESTAMPTZ`（タイムゾーン付き）を使用する。

Supabase はUTC管理のため、表示時にフロントエンドで Asia/Tokyo へ変換する。

---

## 文字コード

UTF-8

# 20. Supabase固有考慮事項

## 20.1 スキーマ

アプリ用テーブルは `public` スキーマに配置する。

---

## 20.2 RLS有効化

全テーブルで `ALTER TABLE ... ENABLE ROW LEVEL SECURITY;` を実行する。

---

## 20.3 API利用方針

| 操作           | 利用API                  |
| ------------ | ---------------------- |
| 基本CRUD       | Supabase Auto REST API |
| 集計・複雑クエリ     | Supabase RPC（DB関数）     |
| リアルタイム更新（将来） | Supabase Realtime      |

---

## 20.4 クライアントキー管理

GitHub Pages からの接続は `anon key` を使用する。

`anon key` は公開されるため、RLS により必ずアクセス制御を実施する。

`service_role key` はフロントエンドに含めない。
